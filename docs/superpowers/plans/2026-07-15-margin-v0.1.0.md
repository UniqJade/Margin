# Margin v0.1.0 Personal Reading Release Plan

## Outcome

Ship a source-only, personal-first `v0.1.0` whose verified happy path is English
reading in Apple Books on macOS. Margin must turn a user-selected passage of two
to four sentences into natural Simplified Chinese, preserve material ambiguity,
and offer an optional semantic alignment view without producing a second,
conflicting translation.

The public repository is a showcase and development record. It does not ship a
public binary or promise end-user installation support. iOS and iPadOS remain
experimental in this release.

## Locked Product Decisions

- macOS is the primary and verified platform.
- `deepseek-v4-flash` is the only certified provider/model for `v0.1.0`.
- DeepSeek is the only prominent provider in Settings.
- A collapsed Advanced section retains an unverified OpenAI-compatible endpoint
  and model interface for future use.
- Margin never silently changes providers or sends the same selection to multiple
  providers.
- Passage translation is optimized for selections of two to four sentences.
- A nuance note appears only when ambiguity materially changes meaning, tone,
  reference, or relationship.
- Word lookup is frozen except for regression fixes. Phrase lookup, OCR, offline
  dictionaries, additional language pairs, learning systems, and public binary
  distribution are out of scope.

## Workstream 1: Passage Contract and DeepSeek Output

### Data model

1. Add a validated semantic alignment block containing ordered source sentence
   indexes and one natural Chinese segment.
2. Make the passage's complete natural translation derive from the ordered
   Chinese segments so the full and aligned views cannot disagree.
3. Retain backward-compatible decoding for cached legacy passage results that
   contain only `translation`, `nuanceNote`, and `literalGloss`.
4. Define a stable presentation identity that includes alignment content.

### Request and validation

1. Update the passage JSON contract and DeepSeek JSON-object prompt.
2. Segment the selected English passage locally and give the provider numbered
   source sentences.
3. Require alignment blocks to cover every source sentence index exactly once,
   in ascending order, while allowing a block to cover multiple adjacent
   sentences.
4. Reject missing, duplicated, overlapping, reordered, or out-of-range indexes.
5. Permit at most one constrained repair request, preserving the current
   injection-resistant quoted-data boundary.
6. If legacy translation output is otherwise valid, retain a safe natural-only
   result rather than inventing alignment locally.

### Passage UI

1. Add a localized `Natural Translation / Semantic Alignment` mode switch.
2. Reset every new result to Natural Translation.
3. Replace the reading body when switching modes; do not append duplicate text.
4. Render each alignment block as a compact English/Chinese semantic pair.
5. Keep the existing action bar fixed and preserve its Copy, Speak, Save, and
   Retry icons.
6. Copy and Speak use the currently visible mode.
7. Preserve the 280--620 pt panel bounds, top anchoring, adaptive height,
   appearance modes, orange accent, and reduced-motion behavior.

### Tests

- Codable round trips and legacy passage decoding.
- Source segmentation and ordered coverage validation.
- Duplicate, omitted, reordered, and out-of-range index rejection.
- Repair behavior and prompt-injection fixtures.
- Presentation identity changes when alignment changes.
- New results reset to Natural Translation.
- Visible-mode copy/speak content is deterministic.

## Workstream 2: Bounded Cache and Intentional History

### Cache

1. Replace the unbounded result dictionary with versioned cache entries that
   include last-access metadata and encoded size accounting.
2. Set a hard target budget of 10 MB and evict least-recently-used entries after
   insertion.
3. Never evict the just-inserted entry unless that single entry exceeds the
   budget; oversized single entries are not cached.
4. Expose current approximate cache usage and a clear-cache action.
5. Keep cache persistence best-effort so a cache failure cannot fail a lookup.

### History and migration

1. Stop recording every successful lookup as a visible history entry.
2. Persist an entry only when the user explicitly presses Save.
3. Saving an unsaved outcome creates the durable entry; unsaving removes it from
   the visible saved collection.
4. Before first migration, atomically copy the legacy history file to a local,
   non-synchronized backup under Margin's Application Support directory.
5. Migrate only legacy entries whose `isSaved` value is true.
6. Version the migration and make it idempotent. Never remove or replace the
   source file when backup or decode fails.
7. Expose separate Clear Cache and Clear Saved Items controls with confirmation.

### Tests

- LRU touch order, eviction, size accounting, oversized entries, and clear.
- Cache compatibility and corrupt-file fallback.
- Uncollected lookup, explicit save, unsave, and repeated-save behavior.
- Backup-before-migration ordering, saved-only migration, idempotency, partial
  failure, corrupt legacy data, and interrupted writes.

## Workstream 3: Localization, Onboarding, and Provider Hierarchy

1. Add standard English and Simplified Chinese string catalogs/resources for all
   user-visible labels, errors, accessibility labels, Help text, Settings,
   History, and onboarding.
2. Keep `Margin` untranslated and retain the English brand line where intended.
3. Add one-page first-run setup:
   - explain the personal reading workflow;
   - apply the DeepSeek endpoint and `deepseek-v4-flash` model;
   - save and test the device-only API key with harmless input;
   - teach `Control-Option-M`;
   - explain Accessibility without requesting it.
4. Mark onboarding complete only after explicit completion; allow it to be opened
   again from Settings.
5. Keep Accessibility prompting exclusively in the shortcut capture path.
6. Show DeepSeek as the primary Settings path. Put custom compatible endpoint and
   model fields in a collapsed Advanced section labelled unverified.
7. Add English and Simplified Chinese hosted UI tests for key screens and state
   transitions.

## Workstream 4: Blind Evaluation Tool

1. Add a local static evaluation app under `Evaluation/` with no server, account,
   analytics, or network dependency.
2. Import a prepared local JSON dataset, randomize Apple and DeepSeek output as
   A/B per item, and keep the source hidden until the session is finalized.
3. Record accuracy, naturalness, ambiguity handling, and overall preference.
4. Save progress in browser-local storage and export detailed JSON/CSV plus an
   aggregate summary.
5. Check in only public-domain, self-authored, or explicitly licensed source
   samples. Keep copyrighted private samples in ignored local files.
6. Document author-only bias and the preparation/blinding limitations.

### Release gate

- 30--40 samples split evenly across biography/history, fiction/dialogue,
  news/general nonfiction, and idiom/ambiguity/complex syntax.
- DeepSeek naturalness preference at least 60%.
- DeepSeek accuracy equal to or better than Apple on at least 90%.
- At most one major semantic error.
- Prompt tuning uses a development subset; the final gate uses held-out samples.
- Failure delays `v0.1.0`; thresholds are not reduced to meet a date.

## Workstream 5: Brand, Public Documentation, and Signing Configuration

1. Create a Margin AppIcon based on an abstract page with a restrained warm-orange
   margin annotation. Avoid generic book, magnifier, or translation-bubble marks.
2. Provide light/dark-safe icon renditions and verify all required Apple sizes.
3. Keep the local project directory and Xcode target names as `BooksTranslator`;
   use `margin-reader` only as the intended GitHub repository name.
4. Make `README.md` the English product page and add `README.zh-CN.md` as a full
   Simplified Chinese counterpart.
5. Move detailed build/signing instructions to `docs/building.md` and evaluation
   methodology/results to `docs/evaluation.md`.
6. Clearly label macOS verified, iOS/iPadOS experimental, source-only BYOK, and no
   public binary support.
7. Move the personal development team configuration to ignored
   `Local.xcconfig`; check in only `Local.xcconfig.example`.
8. Update project generation, install, and verification scripts to read the local
   config while preserving the owner's existing bundle IDs, TeamIdentifier, and
   installed path.
9. Do not publish `claude_work.md`, API keys, certificates, private keys,
   provisioning profiles, private evaluation inputs, build products, or local
   design-session state.

## Repository Slimming

1. Record before/after disk usage.
2. Remove ignored, reproducible `.build` products and stale DerivedData under the
   repository.
3. Keep source, tests, Git history, the small design prototypes, local Claude
   permission record, and useful screenshots until their public/private role is
   decided.
4. Keep generated products in ignored `.noindex` paths and provide a safe cleanup
   script or documented command.
5. Rebuild only the artifacts required for verification, then report the final
   steady-state footprint and the optional reclaimable build footprint.

## Parallel Execution

- Primary agent: passage contract, DeepSeek provider, integration, final builds,
  signing, and release audit.
- Storage agent: bounded cache, saved-only history, migration, and tests.
- Localization agent: string resources, onboarding, provider hierarchy, and tests.
- Public/evaluation agent: evaluator, documentation, local signing config, public
  hygiene, and icon asset preparation.

Agents must avoid editing the same files concurrently. Shared project generation,
package manifests, README integration, and final verification remain owned by the
primary agent.

## Verification

1. `git diff --check`
2. `xcodegen generate` twice with no second-run drift
3. `swift test`
4. `./scripts/test-mac.sh`
5. iOS Simulator app and Action Extension build
6. Release Mac install and signature verification using the existing stable
   identity
7. Manual Apple Books checks in light/dark/full screen, short/long passage,
   cache hit, offline, invalid key, retry, save/unsave, migration, and first-run
   setup
8. English and Simplified Chinese visual checks
9. Secret scan and ignored-file audit before public `v0.1.0`

## Schedule Target

Target three to five days of wall-clock collaboration. Major implementation is
expected in the first one to two days; remaining time is reserved for integration,
owner visual review, the 60--90 minute blind evaluation, prompt correction, and
held-out revalidation. Quality gates override the schedule.
